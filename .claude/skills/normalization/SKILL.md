# Skill: 정규화/반정규화 판단

## 사용 시점
- ERD 설계 시 정규화/반정규화 결정이 필요할 때
- review-agent가 ERD 검토 시

---

## 정규화 판단 기준

### 언제 정규화하는가?
- 같은 데이터가 여러 테이블에 반복되는 경우
- 데이터 변경 시 여러 곳을 수정해야 하는 경우
- 삽입/삭제 이상이 발생하는 구조인 경우

### 언제 반정규화를 고려하는가?
아래 조건을 **모두** 만족할 때만 반정규화를 적용한다:
1. 해당 조회가 매우 자주 발생하는가?
2. 집계/JOIN 비용이 실측상 문제가 되는가?
3. 반정규화 후 데이터 불일치 가능성이 없는가?
4. 반정규화 결정을 docs/02-erd-logical.md에 기록할 수 있는가?

### Passly 반정규화 적용 사례

#### exam_sessions.correct_count
```
정규화 원칙: user_attempts COUNT로 계산
반정규화 이유:
  - 대시보드 진입마다 집계 쿼리 발생
  - 시험 완료 후 correct_count는 변하지 않음 (불변 값)
  - 데이터 불일치 가능성 없음
```

## 정규화 검토 체크리스트

### 1NF
- [ ] 각 컬럼이 단일 값인가?
- [ ] 반복 그룹이 없는가?

### 2NF (복합 PK인 경우만)
- [ ] 모든 비키 컬럼이 전체 PK에 종속되는가?
- [ ] 부분 종속이 없는가?

### 3NF
- [ ] 이행 종속이 없는가? (PK → A → B 구조 금지)
- [ ] 비키 컬럼이 다른 비키 컬럼에 종속되지 않는가?

### 반정규화 적용 시 문서 형식
```markdown
## 반정규화 결정: [테이블명.컬럼명]
- 배경: [왜 집계가 필요한가]
- 선택: [어떻게 반정규화했는가]
- 근거: [왜 정당한가]
- 트레이드오프: [어떤 단점이 있는가]
- 정합성 유지 방법: [어떻게 동기화하는가]
```
